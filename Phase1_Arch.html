API GATEWAY — GraphQL Federation + Auth
🔮
Apollo Federation Gateway Node.js OSS
Single /graphql endpoint. Composes 4 subgraphs (Catalog, Compute, Quality, Governance). DataLoader batching. Schema introspection for auto-discovery.
Apollo Federation Gateway
Federated Subgraphs
● Catalog → Catalog Service (Java) — definitions, search, lineage
● Compute → Computation Service (Python) — metric values, backfills
● Quality → Quality Service (Python) — scores, check results
● Governance → Governance Service (Java) — approvals, diffs, audit
Open-Source Stack
● Apollo Server 4 (MIT License)
● Apollo Gateway for federation routing
● DataLoader — batching + caching service calls
● GraphQL Shield — declarative permission layer
🔐
Auth & RBAC OSS
OAuth2 / JWT + Open Policy Agent (OPA). Field-level auth, row-level security, query audit logging. AD/LDAP integration.
Authentication & Authorization
Authentication
● OAuth2 / OpenID Connect for user auth
● JWT tokens with metric-scope claims
● Active Directory / LDAP integration (bank SSO)
● Service accounts via mTLS for system consumers
Authorization (OPA — Open Source)
● Field-level: sensitive metrics restricted by role
● Row-level: regional managers scoped to their region
● Tier-based: who can author/approve at each tier
Audit Trail
● Every query logged: caller, timestamp, metric, params
● Persisted to SQL Server audit_log table
Resolvers delegate to microservices (REST / gRPC)
SERVICE LAYER — Java + Python Microservices (all open-source frameworks)
☕
Catalog Service Java OSS
Metric CRUD, versioning, full-text search, algorithmic SQL deduplication. Reads/writes metric definitions to SQL Server.
Catalog Service — Java / Spring Boot
Responsibilities
● Metric definition CRUD with optimistic locking
● Full-text search via Elasticsearch sidecar
● Algorithmic deduplication — canonical SQL comparison
● Composite metric detection & dependency graph
● Immutable version history management
Key APIs
● POST /metrics — create definition
● PUT /metrics/{key}/versions — propose change
● GET /metrics/search?q= — catalog search
● GET /metrics/{key}/lineage — dependency graph
Open-Source Stack
● Java 17+ / Spring Boot 3.x (Apache 2.0)
● Spring Data JPA → SQL Server via mssql-jdbc
● Apache Calcite — SQL parsing & column lineage (Apache 2.0)
● Elasticsearch — search index (SSPL / Apache 2.0 via OpenSearch)
🐍
Computation Service Python OSS
SQL generation from definitions, batch computation against SQL Server, scheduling, backfill engine for historical recomputation.
Computation Service — Python
Responsibilities
● Generate executable SQL from metric definitions + runtime params
● Execute queries directly against SQL Server
● Write pre-computed results back to SQL Server computed store
● Backfill engine — recompute historical data on definition change
● Scheduled batch runs aligned with source data freshness
Key APIs
● POST /compute/{key} — on-demand computation
● POST /compute/batch — scheduled bulk run
● POST /compute/{key}/backfill — historical recompute
Open-Source Stack
● Python 3.11+ / FastAPI (MIT)
● SQLAlchemy + pyodbc → SQL Server connectivity
● Jinja2 — SQL template rendering
● APScheduler or Celery — job scheduling (Phase 1)
🔍
Quality Service Python OSS
Freshness, completeness, reconciliation, tolerance monitoring. All checks run queries against SQL Server source tables.
Quality Service — Python
Quality Checks (all against SQL Server)
● Freshness — source tables within declared SLA window
● Completeness — NULL / missing record detection
● Cross-source reconciliation — compare digital mart vs EDW totals
● Duplicate detection — PK & composite key scans
● Tolerance monitoring — period-over-period anomaly detection
Open-Source Stack
● Python 3.11+ / FastAPI (MIT)
● Great Expectations (Apache 2.0) — quality framework
● SQLAlchemy + pyodbc → SQL Server
● Prometheus client — quality SLA metrics (Apache 2.0)

STORAGE LAYER — SQL Server (single backend) + supporting stores
🗄️
Metadata Store SQL Server
Metric definitions, versions, dimensions, ownership, approval records, quality results, audit logs.
SQL Server — Metadata Store
Core Tables
● metric_definitions — key, name, tier, status, owner
● metric_versions — version_id, sql_logic, effective_date, author
● metric_dimensions — dimension specs, allowed_values
● source_views — unified view definitions
● approval_records — change requests, actions, comments
● quality_results — check outcomes, scores, timestamps
● audit_log — all actions with actor + timestamp
Design Decisions
● Immutable version history — append only, no updates
● Optimistic locking (rowversion) for concurrent edits
● Partitioned audit log by month for query performance
📦
Computed Metric Store SQL Server
Pre-computed results, materialized aggregations, dimension rollup cache. Columnstore indexes for analytical queries.
SQL Server — Computed Metric Store
Core Tables
● metric_results — key, period, dimension_values (JSON), value, version
● aggregation_cache — daily → monthly → quarterly rollups
Design
● Clustered columnstore index on metric_results
● Partitioned by period (monthly) for fast range scans
● Results tagged with definition_version for traceability
📝
Definition Repo Git OSS
Version-controlled SQL definitions. Native diffing, branch-per-change-request, merge on approval. Gitea or GitLab CE self-hosted.
⚡
Metric Cache Redis OSS
Hot metric cache for frequently queried results. TTL-based eviction. Rate limiting for API gateway. Redis 7 (BSD License).
Source view abstraction — services query source tables in SQL Server

Phase 1 Data Flow Paths
Query Path
Consumer → GraphQL Gateway → Service → SQL Server (computed store) → Redis Cache → Response
Authoring Path
BankMetric UI → Catalog Service → SQL Server (metadata) + Git Repo → Governance Service → Approval Workflow
Computation Path
Scheduler → Compute Service → SQL Server (digital data mart source) → SQL Server (computed store) → Redis invalidation
Quality Path
Cron → Quality Service → SQL Server (source table checks) → SQL Server (quality_results) → Alert if degraded
