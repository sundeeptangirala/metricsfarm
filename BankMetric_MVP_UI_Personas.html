BankMetric Store

MVP UI FEATURES

Persona-driven feature map for the Phase 1 MVP. Each persona shows their goals, pain points, primary workflows, and the specific UI features they need. Features are prioritized as MUST HAVE, SHOULD HAVE, or NICE TO HAVE.

Personas & Their UI Features
📝
Metric Author
Data Analyst / Data Engineer
~15 users in Phase 1
Goal: Define new metrics or modify existing ones with confidence that the definition is unambiguous, not duplicated, and will go through proper review before anyone consumes it.
Pain today: Writes SQL in personal scripts or Excel. No way to know if someone else already defined the same metric differently. Changes go live with no review.
Primary Workflow
🔍 Search catalog
→
✏️ Author / edit definition
→
🧪 Preview SQL output
→
🔄 See before/after diff
→
🚀 Submit for review
→
📬 Get notified on approval
UI Features Needed
✏️
Metric Definition Form
Structured form: metric key, display name, description, tier, owner, source view, dimensions. Validates snake_case naming.
MUST HAVE
💻
SQL Editor with Syntax Highlighting
Monaco or CodeMirror editor for metric SQL. Syntax highlighting, auto-complete for table/column names from source views.
MUST HAVE
🧪
SQL Preview / Dry Run
Execute the metric SQL against SQL Server with sample parameters and show results before submitting. Catches errors early.
MUST HAVE
🔍
Duplicate Detection Alert
On save/submit, run deduplication check. Show similar existing metrics with side-by-side comparison if near-match found.
SHOULD HAVE
🔄
Before / After Diff View
Side-by-side SQL diff when editing an existing metric. Color-coded additions (green) and removals (red). Auto-generated on edit.
MUST HAVE
💾
Save as Draft
Save work-in-progress definitions without triggering review. Draft status visible only to the author and their team.
MUST HAVE
✅
Metric Reviewer
Data Steward / Senior Analyst / LOB Lead
~8 users in Phase 1
Goal: Review proposed metric changes quickly and confidently — understand what changed, why, and what downstream impact it has — then approve or reject with documented reasoning.
Pain today: Reviews happen over email or Slack with screenshots of SQL. No structured diff, no impact visibility, no audit trail of approval decisions.
Primary Workflow
📬 Receive review notification
→
🔄 View before/after diff
→
📊 Check sample data comparison
→
⚠️ Review downstream impact
→
✅ Approve / ❌ Reject / 💬 Comment
UI Features Needed
📋
Approval Queue Dashboard
Inbox-style list of pending change requests assigned to the reviewer. Sorted by submission date, filterable by tier and metric domain.
MUST HAVE
🔄
Side-by-Side Diff Panel
Full SQL diff with line-level highlighting. Metadata changes (description, dimensions, owner) shown alongside SQL changes.
MUST HAVE
📊
Sample Data Comparison
Run both old and new SQL against recent data. Show results side-by-side so reviewer can see numeric impact of the change.
MUST HAVE
⚠️
Downstream Impact Panel
List all dashboards, reports, and API consumers that use this metric. Highlight breaking vs. non-breaking changes.
SHOULD HAVE
💬
Inline Comments
Add comments on specific lines of the SQL definition. Threaded replies between author and reviewer. Persisted for audit.
SHOULD HAVE
✅
Approve / Reject / Request Changes
Three-action buttons with mandatory comment on reject. Approval triggers version publish + downstream notification.
MUST HAVE
📊
Metric Consumer
Business Analyst / Product Manager / Ops
~50+ users in Phase 1
Goal: Find the right metric quickly, understand exactly what it measures, trust the data quality, and get the number into their dashboard or analysis without writing SQL.
Pain today: Asks three people "which table has session counts?" and gets three different answers. Doesn't know if the number in their dashboard is stale or wrong.
Primary Workflow
🔍 Search / browse catalog
→
📖 Read definition & metadata
→
🟢 Check quality score
→
📋 Copy GraphQL query
→
📊 Use in Tableau / Jupyter
UI Features Needed
🔍
Metric Catalog with Search
Full-text search across metric names, descriptions, and keys. Filter by tier, status, quality score, owner, and domain.
MUST HAVE
📖
Metric Detail Page
Everything about a metric: description, SQL logic, source view, dimensions, owner, version history, quality score, last computed timestamp.
MUST HAVE
🟢
Quality Score Badge
Traffic-light indicator (green/amber/red) on every metric. Click to see which checks passed/failed and when last run.
MUST HAVE
📋
Copy-to-Clipboard Query Builder
Auto-generate GraphQL query for the selected metric with chosen dimensions. One-click copy for pasting into Jupyter, Tableau, or cURL.
SHOULD HAVE
📜
Version History Timeline
Visual timeline of all definition changes. Click any version to see who changed what, when, and why. Compare any two versions.
SHOULD HAVE
🔔
Subscription / Watch
Follow a metric to get notified when its definition changes or quality degrades. Email or Slack notifications.
NICE TO HAVE
⚙️
Platform Admin
Data Engineer / Metric Store Team
~3–5 users in Phase 1
Goal: Keep the metric store healthy — monitor computation jobs, manage source view mappings, configure quality checks, and handle access control. Ensure the platform serves all other personas reliably.
Pain today: No centralized view of metric health. Troubleshooting a bad number requires tracing through multiple SQL scripts, Airflow DAGs, and dashboards manually.
Primary Workflow
📊 Monitor job health
→
🔴 Investigate quality alerts
→
🗄️ Manage source views
→
🔐 Configure access control
→
🔄 Trigger backfills
UI Features Needed
📊
Computation Job Monitor
Dashboard showing batch job status: running, succeeded, failed. Execution time, last run, next scheduled run. Click to see logs.
MUST HAVE
🟢
Quality Health Dashboard
Aggregate quality view across all metrics. Sortable by score. Drill-down to individual check results, failure details, and source freshness.
MUST HAVE
🗄️
Source View Manager
CRUD for source views — map SQL Server tables/views to the abstraction layer. Validate connectivity and column availability.
MUST HAVE
🔐
Access Control Panel
Manage roles and permissions per metric or tier. Map AD groups to metric access levels. View audit log of access changes.
SHOULD HAVE
🔄
Backfill Trigger
Manually trigger historical recomputation for a metric across a date range. Monitor progress and preview results before promoting.
SHOULD HAVE
📈
Platform Usage Analytics
Track API query volume, most-queried metrics, active users, adoption trends. Proves ROI to leadership.
NICE TO HAVE
👔
Executive Sponsor
CDO / VP Analytics / LOB Head
~5 users in Phase 1
Goal: See that metrics are governed, trustworthy, and widely adopted. Needs a quick pulse on platform health without diving into details. Wants proof the investment is paying off.
Pain today: Gets asked "which number is right?" in board meetings. Has no visibility into whether metric governance is actually happening or just aspirational.
Primary Workflow
📊 View adoption dashboard
→
🟢 Check overall quality health
→
📋 Review governance activity
→
📈 Track ROI metrics
UI Features Needed
📊
Executive Summary Dashboard
Single-page view: total metrics governed, quality distribution (green/amber/red), active consumers, approval throughput, adoption trend.
SHOULD HAVE
📈
Adoption & ROI Scorecard
Metrics governed vs. ungoverned, time saved, reconciliation incidents avoided, query volume growth. Exportable for board reporting.
NICE TO HAVE
📋
Governance Activity Log
Recent approvals, rejections, and pending items. Shows governance is active, not dormant. Filterable by LOB and tier.
SHOULD HAVE

MVP Scope Summary
14 MUST-HAVE features across 5 personas — this is the MVP cut line. Ship these first. They cover the core loop: author a metric → review the change → approve it → consume it → monitor quality.
9 SHOULD-HAVE features — build these in the second sprint cycle. They add depth (inline comments, impact analysis, version timeline) but aren't blockers for first value.
3 NICE-TO-HAVE features — backlog for Phase 1 hardening or Phase 2. Notifications, usage analytics, and ROI scorecards prove long-term value but aren't needed at launch.
